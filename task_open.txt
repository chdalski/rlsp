- [x] YAML parsing with diagnostic publishing
- [x] Hover support

## Remaining Tasks (in order)

### Task 1: Code Actions
Quick fixes and refactoring actions.

**What to build:**
- New module `src/code_actions.rs` with pure function `code_actions(text, range, diagnostics) -> Vec<CodeAction>`
- Actions to implement:
  1. **Convert flow to block style** — `{a: 1}` -> multi-line block mapping
  2. **Convert block to flow style** — multi-line -> `{a: 1}`
  3. **Tab to space conversion** — fix indentation issues
  4. **Delete unused anchor** — remove `&name` when marked as unused by validator
  5. **Convert quoted string to boolean** — `"true"` -> `true`
  6. **Convert string to block scalar** — long strings to `|` or `>` style
- Server integration: add `code_action_provider` capability and handler

**Acceptance criteria:**
- Code actions appear for relevant contexts
- Applying an action produces valid YAML
- Delete unused anchor only appears when diagnostic is present
- Style conversions preserve semantics

### Task 3: Schema Support (Large - Multi-Part)
JSON Schema integration for validation, completion, and hover.

**Part A: Schema Loading and Association**
- New module `src/schema.rs` for schema types and loading
- Schema association by:
  1. File pattern matching (e.g., `*.k8s.yaml` -> kubernetes schema)
  2. Modeline comment (`# yaml-language-server: $schema=<url>`)
  3. Workspace settings
- Fetch schemas from URLs (schemastore.org) with caching
- Parse JSON Schema (Draft-04, Draft-07 at minimum)

**Part B: Schema Validation**
- Extend diagnostic publishing to include schema validation errors
- Validate required properties, types, enum values, additionalProperties
- Produce clear diagnostic messages with property paths

**Part C: Schema-Driven Completion**
- Extend `completion.rs` to accept schema context
- Suggest property names from schema
- Suggest enum values from schema
- Include descriptions in completion items

**Part D: Schema-Driven Hover**
- Extend `hover.rs` to accept schema context
- Show property description from schema
- Show type, default value, examples from schema

**Acceptance criteria:**
- Schemas load from URLs and cache locally
- Modeline association works (`# yaml-language-server: $schema=...`)
- Validation errors appear as diagnostics
- Completion suggests schema-defined properties and enum values
- Hover shows schema descriptions

**Dependencies:** May require an HTTP client crate (e.g., `reqwest` or `ureq`) for fetching schemas — user approval needed.

### Task 4: On-Type Formatting

**What to build:**
- Server integration: add `document_on_type_formatting_provider` capability
- Trigger characters: `\n` (newline)
- Auto-indent logic:

## Architecture Notes
- Pure function design: each feature module exports a pure function (text + position -> result)
- saphyr 0.0.6 for YAML parsing (migrated from yaml-rust2)
- MarkedYamlOwned for AST nodes with span info (used by selection ranges)
- tower-lsp v0.20 with "proposed" features
- tokio for async runtime
