- **`string_to_block_scalar` doubles anchors in output** â€” When converting `description: &myanchor "long string"` to a block scalar, `format_subtree` re-emits `&myanchor` in the edit's `new_text` while the edit's `range.start` lands after the source `&myanchor ` prefix (the parser's scalar `loc` excludes the anchor). Result: `description: &myanchor &myanchor |\n  long string`. Surfaced during the code-action fixture port (commit `94d5cfc`) â€” the original inline test asserted `result.contains("&myanchor")` which was satisfied by either one or two occurrences, so the doubling shipped under a loose assertion. The fixture `tests/fixtures/code_actions/block-scalar-preserve-anchor.md` currently codifies the doubled output as expected; once fixed, that fixture's `Expected-Document` updates to a single `&myanchor`. Fix the production code path; do not loosen the fixture format to mask this.

- **Validator severity inversion: resolve at the producer, not in `parse_and_publish`** â€” `validate_flow_style` and `validate_duplicate_keys` (`rlsp-yaml/src/validation/validators.rs:218`, `:567`) construct diagnostics with hardcoded `WARNING`/`ERROR`. `rlsp-yaml/src/server.rs:476â€“496` then walks the returned vec and overwrites `diag.severity` based on the user's `flowStyle` and `duplicateKeys` settings. Same root shape as the code-action user-format-config bug (producer hardcodes a default; configuration applied â€” or not â€” elsewhere), but here the override exists; it's just detached from the producer. Why it's a smell, not just stylistic: (1) the validators look unconditional in isolation â€” the truth that severity is configurable lives only in `server.rs`, so a future change flipping a validator's default to ERROR would silently no-op the `== "error"` check; (2) construct-then-rewrite is `Loop+Assignment+Conditional` mass for a value the producer already had access to; (3) it's an asymmetric precedent â€” the other validators (`unused_anchors`, `key_ordering`, `custom_tags`, `yaml11_compat`, schema) bake severity in with no override path, so the next configurable severity has only this two-step pattern to copy. Fix: thread the relevant settings into the validator (or pass a category and resolve once at a single boundary) so user-facing severity is determined at the source. Sequencing: schedule after `2026-04-27-code-action-respect-user-format-config.md` lands â€” the dispatch redesign there (plumbing settings into feature handlers) cleanly absorbs this. No user-facing behavior change; this is a code-cleanup follow-up.
