- **`block_to_flow` policy enforcement against `formatEnforceBlockStyle`** — When the user sets `formatEnforceBlockStyle: true`, the workspace formatter rewrites all flow-style collections back to block style on save. The `block_to_flow` code action contradicts this policy: applying it produces flow output that the next save reverts. Open question: should `block_to_flow` be suppressed (not offered) when `formatEnforceBlockStyle: true`, or should it remain offered with a UI hint, or is the conflict acceptable as user-driven? Surfaced during the code-action user-format-config plan audit (`.ai/plans/2026-04-27-code-action-respect-user-format-config.md`, in Decisions/Non-Goals). Separate plan needed — the question is policy/UX, not a correctness bug. Related but distinct: are there other code actions whose existence violates user formatter policy when settings are non-default?

- **`autoWrapFlowStyle` opt-out for `block_to_flow`** — The code-action user-format-config plan (`.ai/plans/2026-04-27-code-action-respect-user-format-config.md`) sets `block_to_flow`'s default to auto-wrap output to fit `formatPrintWidth`, removing the misleading `(long line)` title suffix. If users later report wanting single-line flow output regardless of length (e.g., for tooling that prefers compact JSON-like flow), add a setting like `autoWrapFlowStyle: false` (default `true`) so the action emits unwrapped flow when the user opts out. Defer until evidence shows demand — pre-emptively shipping the setting risks configuration sprawl across all code actions. Bias: do nothing unless a user explicitly asks for it.

- **Document rlsp-yaml ↔ prettier (and other formatters) interop** — The code-action user-format-config plan (`.ai/plans/2026-04-27-code-action-respect-user-format-config.md`) made code-action output use rlsp-yaml's `formatPrintWidth`. Many users run rlsp-yaml as their LSP server but prettier (or a different formatter) as their format-on-save formatter. In that setup the user maintains two parallel print-width settings: rlsp-yaml's `formatPrintWidth` (controls code-action output shape) and prettier's `printWidth` (controls save-time reformatting). Both default to 80, so most users won't notice — but a customized one without the other produces mid-edit visual jitter (code action wraps for 80; save reformats to 100 or vice versa). Documentation-only follow-up: add a section to `rlsp-yaml/docs/configuration.md` explaining the interaction, listing which rlsp-yaml settings have prettier equivalents (`formatPrintWidth`/`printWidth`, `formatSingleQuote`/`singleQuote`, `formatBracketSpacing`/`bracketSpacing`), and recommending users keep them aligned when both formatters are active. **Out of scope:** automatic cross-formatter config awareness (e.g., reading `.prettierrc` for fallback values) — that's a much heavier change and invites a "which other prettier knobs do we honor?" rabbit hole. Start with documentation; consider richer interop only if user demand emerges.

## Open: rlsp-yaml-parser

- **Strict c-printable enforcement on literal stream characters (§5.1)** — The parser enforces `is_c_printable` only at the numeric-escape decoder (`lexer/quoted.rs:580`) and at a few pinpoint cases (NUL in trailing comments, BOM in document body, NUL/BOM in plain-scalar suffixes). Literal non-printable bytes pass through silently in plain scalars, single/double-quoted scalar literal content (between escapes), block scalar content, and standalone comment lines. Per YAML 1.2.2 §5.1 ("A YAML stream consists only of c-printable characters") and §5.7 ("non-printable characters are not available" outside double-quoted), this is non-conformant — input containing literal non-printables is invalid YAML by spec. The conformance doc currently labels [1] c-printable as "Conformant" but cites only the predicate definition (`chars.rs:14-26`); enforcement is missing. Audit confirms `Lenient` (commit 6f0ec6d).
- **`nb-char` enforcement gap (§5.4)** — Production [27] `nb-char ::= c-printable - b-char - c-byte-order-mark` is unenforced for the same root cause as [1]: the `nb-char` predicate is partially synthesized via the line splitter (which excludes line breaks) but the `c-printable` constraint is not enforced on slice content. The line splitter at `lines.rs:130-132` correctly handles `b-char` exclusion but does not gate non-printable bytes within line content. Fixing [1]'s enforcement gap propagates here. Surfaced by the conformance audit `2026-04-30-phase1-bnf/reconciliation-§5.md`. Doc currently mislabels as Conformant.
- **`ns-char` enforcement gap (§5.5)** — Production [34] `ns-char ::= nb-char - s-white` predicate at `chars.rs:67-74` is partially used (plain-scalar starter, unrecognized-line first-char) but not enforced on plain-scalar bodies or quoted-scalar bodies. Same root cause as [1]/[27]. Surfaced by `2026-04-30-phase1-bnf/reconciliation-§5.md`. Doc currently mislabels as Conformant.
- **`s-flow-line-prefix(n)` indent vs separation enforcement (§6.3)** — Production [69] requires `s-indent(n) s-separate-in-line?` (n SPACE characters first, then optional whitespace allowing tabs). The implementation's `trim_start_matches([' ', '\t'])` at `lexer/quoted.rs:107` and `:276` does not enforce that the first n characters are spaces — a continuation line with leading tabs is accepted as if tabs counted toward indent. Fix: distinguish indent (n spaces required) from separation (whitespace allowed) in the continuation prefix logic. Surfaced by `2026-04-30-phase1-bnf/reconciliation-§6.md`.
- **Comment body BOM filtering (§6.6, [75] c-nb-comment-text)** — Comment body slice at `lexer/comment.rs:50-51` (`text: &line.content[text_start..]`) does not exclude BOM (`\u{FEFF}`); a BOM character in a comment body is retained verbatim, contradicting `nb-char`'s exclusion of c-byte-order-mark. Fix propagates from [27] gap; if c-printable enforcement is added at content slicing, BOM filtering follows. Surfaced by `2026-04-30-phase1-bnf/reconciliation-§6.md`.
- **Directive name + parameter validation (§6.8, [84]/[85])** — `ns-directive-name` and `ns-directive-parameter` blobs are not validated against `ns-char+` in `event_iter/directives.rs`. The parser accepts directives with non-`ns-char` bytes in the name or parameter (e.g., `%FOO bad\x00content`). The spec licenses unknown directives via "should ignore … with appropriate warning," but ignoring does not mean accepting malformed body shapes. Decision needed: validate name/parameter against `ns-char+`, OR document the laxity as an intentional implementation choice for unknown directives. Surfaced by `2026-04-30-phase1-bnf/reconciliation-§6.md`.
- **Tag prefix validation (§6.9.1, [93]/[94]/[95])** — `ns-tag-prefix`, `c-ns-local-tag-prefix`, and `ns-global-tag-prefix` validation in `event_iter/directives.rs` rejects only ASCII control characters and DEL, not the full `ns-uri-char` / `ns-tag-char` constraints the spec requires. The conformance doc itself notes this leniency in adjacent prose at line 968 yet labels these "Conformant" inconsistently. Fix: validate prefix bytes against `ns-uri-char` per the spec; or relax validation explicitly with a documented rationale. Surfaced by `2026-04-30-phase1-bnf/reconciliation-§6.md`.
- **Empty suffix in shorthand tags (§6.9.1, [99] c-ns-shorthand-tag)** — Production requires `c-tag-handle ns-tag-char+` (non-empty suffix). Implementation in `event_iter/properties.rs:170-176` and `:200-203` explicitly accepts empty suffixes — `!!` and `!handle!` parse as valid shorthand tags. Code includes a comment documenting this acceptance. Fix: require non-empty suffix per the spec, or formally accept the deviation in a documented design note. Surfaced by `2026-04-30-phase1-bnf/reconciliation-§6.md`.
