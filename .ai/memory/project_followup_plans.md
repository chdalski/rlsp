- **`string_to_block_scalar` doubles node properties (anchors and user tags) in output** — When converting `description: &myanchor "long string"` to a block scalar, `format_subtree` re-emits `&myanchor` in the edit's `new_text` while the edit's `range.start` lands after the source `&myanchor ` prefix (the parser's scalar `loc` excludes both `anchor_loc` and `tag_loc`). Result: `description: &myanchor &myanchor |\n  long string`. Surfaced during the code-action fixture port (commit `94d5cfc`) — the original inline test asserted `result.contains("&myanchor")` which was satisfied by either one or two occurrences, so the doubling shipped under a loose assertion. The fixture `tests/fixtures/code_actions/block-scalar-preserve-anchor.md` currently codifies the doubled output as expected; once fixed, that fixture's `Expected-Document` updates to a single `&myanchor`. **Same shape applies to user tags**: `description: !mytag "long string"` produces `description: !mytag !mytag |` (verified empirically). Core-schema tags (`!!str`, `!!int`, etc.) are immune by accident — `formatter.rs:540-573` strips user-authored core tags from non-empty scalars as an idempotency rule, so the formatter never emits them in `new_text`. Combined `&a !!str "..."` shows anchor doubled, core tag stripped. Fix shape: clear `meta.anchor`, `meta.tag`, and `meta.tag_loc` on the cloned scalar before calling `format_subtree` (smaller change), or widen the edit range to include `anchor_loc` and `tag_loc` and the separating spaces (lets the formatter's re-emission stand). Fix the production code path; do not loosen the fixture format to mask this.

- **Parser mis-attaches or drops properties on block mapping with combined `&anchor` and user `!tag`** — Two manifestations of the same root cause in `rlsp-yaml-parser`'s property attachment logic. Verified by AST-level probe via `rlsp_yaml_parser::load()` — the wrong AST is produced before the formatter ever runs. Originally mis-attributed as a formatter bug; corrected after tracing `format_yaml`'s output back to the parser's AST.

  **Bug A — anchor migrates to first key (anchor-first order `&anchor !tag`):** `defaults: &base !mytag\n  timeout: 30\n  retries: 3\n` produces an AST with `outer.anchor() = None`, `outer.tag = Some("!mytag")` ✓, and `first_key.anchor = Some("base")` ✗. Round-trip via `format_yaml` is broken: re-parsing the formatted output binds `&base` to the scalar `30` instead of the parent mapping. `block_to_flow` inherits the symptom: `defaults: !mytag { &base timeout: 30, retries: 3 }`.

  **Bug B — user tag dropped (tag-first order `!tag &anchor`):** `defaults: !mytag &base\n  timeout: 30\n  retries: 3\n` produces an AST with `outer.anchor() = Some("base")` ✓ but `outer.tag = "tag:yaml.org,2002:map"` (resolver-injected core tag) and `outer.meta.tag_loc.is_none()`. The user-authored `!mytag` is silently lost.

  **Unaffected cases (regression coverage targets after fix):** Block sequences with combined props (`items: &list !mytag\n  - a\n  - b`) parse correctly. Flow mappings and flow sequences (`{...}`, `[...]`) parse correctly in both orderings.

  **Likely fix neighborhood:** the lexer's property classification — `pending_anchor`/`pending_tag` with `Standalone`/`Inline` variants of `PendingAnchor`/`PendingTag` (see `rlsp-yaml-parser/src/lib.rs:85-126`). The existing `pending_collection_anchor`/`pending_collection_tag` displacement buffers were designed for a different shape (`&node1\n&k1 key: v` — collection property displaced by key property). The combined-property case has two collection-bound properties in sequence on the same line; investigation will confirm whether the second one overwrites the first's pending state asymmetrically, or whether the classification of one of the two properties is wrong.

  **Status:** Needs a dedicated plan with parser AST unit tests asserting `load(...)` produces correct anchor/tag attribution for both orderings on block mappings. The formatter round-trip corpus invariant (now I10, landed in plan `2026-04-28-corpus-invariant-formatter-round-trip.md`) does NOT surface these bugs — both pre-format and post-format ASTs are equally wrong (parser misparses on first parse and again on re-parse), so round-trip equivalence holds. The bugs only show when comparing AST to source semantics, which a round-trip invariant cannot do. Plan shape when picked up: parser fix in `rlsp-yaml-parser/src/lib.rs` property-classification path, plus an AST-shape test sweep across `(anchor only, tag only, anchor+tag in both orders) × (block mapping, block sequence, flow mapping, flow sequence)` to lock down the matrix.

- **Document rlsp-yaml ↔ prettier (and other formatters) interop** — The code-action user-format-config plan (`.ai/plans/2026-04-27-code-action-respect-user-format-config.md`) made code-action output use rlsp-yaml's `formatPrintWidth`. Many users run rlsp-yaml as their LSP server but prettier (or a different formatter) as their format-on-save formatter. In that setup the user maintains two parallel print-width settings: rlsp-yaml's `formatPrintWidth` (controls code-action output shape) and prettier's `printWidth` (controls save-time reformatting). Both default to 80, so most users won't notice — but a customized one without the other produces mid-edit visual jitter (code action wraps for 80; save reformats to 100 or vice versa). Documentation-only follow-up: add a section to `rlsp-yaml/docs/configuration.md` explaining the interaction, listing which rlsp-yaml settings have prettier equivalents (`formatPrintWidth`/`printWidth`, `formatSingleQuote`/`singleQuote`, `formatBracketSpacing`/`bracketSpacing`), and recommending users keep them aligned when both formatters are active. **Out of scope:** automatic cross-formatter config awareness (e.g., reading `.prettierrc` for fallback values) — that's a much heavier change and invites a "which other prettier knobs do we honor?" rabbit hole. Start with documentation; consider richer interop only if user demand emerges.

- **Validator severity inversion: resolve at the producer, not in `parse_and_publish`** — `validate_flow_style` and `validate_duplicate_keys` (`rlsp-yaml/src/validation/validators.rs:218`, `:567`) construct diagnostics with hardcoded `WARNING`/`ERROR`. `rlsp-yaml/src/server.rs:476–496` then walks the returned vec and overwrites `diag.severity` based on the user's `flowStyle` and `duplicateKeys` settings. Same root shape as the code-action user-format-config bug (producer hardcodes a default; configuration applied — or not — elsewhere), but here the override exists; it's just detached from the producer. Why it's a smell, not just stylistic: (1) the validators look unconditional in isolation — the truth that severity is configurable lives only in `server.rs`, so a future change flipping a validator's default to ERROR would silently no-op the `== "error"` check; (2) construct-then-rewrite is `Loop+Assignment+Conditional` mass for a value the producer already had access to; (3) it's an asymmetric precedent — the other validators (`unused_anchors`, `key_ordering`, `custom_tags`, `yaml11_compat`, schema) bake severity in with no override path, so the next configurable severity has only this two-step pattern to copy. Fix: thread the relevant settings into the validator (or pass a category and resolve once at a single boundary) so user-facing severity is determined at the source. Sequencing: schedule after `2026-04-27-code-action-respect-user-format-config.md` lands — the dispatch redesign there (plumbing settings into feature handlers) cleanly absorbs this. No user-facing behavior change; this is a code-cleanup follow-up.
