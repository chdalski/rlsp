## Goal

Close all four open GitHub Dependabot alerts (#5, #6, #15,
#16) against `rlsp-yaml/integrations/vscode/pnpm-lock.yaml`
by pinning patched versions of two transitive dev
dependencies — `fast-uri` and `serialize-javascript` —
through the existing `pnpm.overrides` block in
`rlsp-yaml/integrations/vscode/package.json`. The VS Code
extension is the only consumer of these packages, both are
dev-scope only (not shipped in the published `.vsix`), and
all four advisories have published patched releases. The
fix should not change any application code, only the
lockfile resolution.

## Context

- **Alerts to close** (from
  `gh api repos/:owner/:repo/dependabot/alerts`):
  - **#16** — `fast-uri <= 3.1.1`, high severity, host
    confusion via percent-encoded authority delimiters
    (CVE-2026-6322, GHSA-v39h-62p7-jpjc). Patched in
    `3.1.2`.
  - **#15** — `fast-uri <= 3.1.0`, high severity, path
    traversal via percent-encoded dot segments. Patched in
    `3.1.1`.
  - **#5** — `serialize-javascript <= 7.0.2`, high
    severity, RCE via `RegExp.flags` and
    `Date.prototype.toISOString()`. Patched in `7.0.3`.
  - **#6** — `serialize-javascript < 7.0.5`, medium
    severity, CPU-exhaustion DoS via crafted array-like
    objects. Patched in `7.0.5`.

- **Dependency chains** (from `pnpm-lock.yaml`):
  - `fast-uri@3.1.0` is a transitive dep of `ajv@8.18.0`.
  - `serialize-javascript@6.0.2` is a transitive dep of
    `mocha@11.7.5` (pulled in by `@vscode/test-cli` for
    integration tests).

- **Override approach**: `package.json` already has a
  `pnpm.overrides` block (currently containing
  `"lodash": ">=4.18.0"`). pnpm overrides force resolved
  versions across the dependency graph regardless of the
  ranges declared by intermediate packages. This is the
  standard pnpm pattern for patching transitive
  vulnerabilities ahead of upstream parent-package
  releases.

- **Major-version bump for `serialize-javascript`**: the
  current version is `6.0.2` and the patched line is
  `7.x`. There is no patched `6.x` release —
  serialize-javascript v7 dropped support for older Node
  versions but retained the same public API. Mocha 11
  declares `serialize-javascript: ^6.0.2`; the override
  bypasses that range. Compatibility is verified by
  running the existing VS Code integration test suite
  (`pnpm run test:integration`), which exercises mocha
  end-to-end.

- **`fast-uri` bump is in-range**: 3.1.0 → ^3.1.2 is a
  patch-level move within the same major. `ajv@8.18.0`
  declares `fast-uri: ^3.0.0`, so the override stays
  within the parent's accepted range.

- **Scope is dev-only**: both packages are flagged with
  `scope: development` by Dependabot and are absent from
  the published extension bundle (`esbuild` builds
  `src/main.ts` with `--external:vscode` and no test
  tooling is bundled). The fix has zero runtime impact on
  users of the published extension; the entire blast
  radius is the build/test pipeline.

- **VS Code extension CLAUDE.md** mandates the default
  pnpm store (no `store-dir` override in `.npmrc`).

## Decisions

- **Override versus parent bump:** chose `pnpm.overrides`
  because the patched parent versions of `ajv` and
  `mocha` that pull in fixed transitives are not yet
  released. Overrides land the fix immediately with
  minimal blast radius. The existing `lodash` override in
  the same file establishes this as the project's
  accepted pattern for transitive dep patching.
- **Caret range for `fast-uri`:** `^3.1.2` allows future
  patch and minor releases within the 3.x line, which is
  consistent with the existing `^` ranges throughout
  `package.json`. The lower bound matches the
  `first_patched_version` of the most stringent advisory
  (#16, host confusion).
- **Caret range for `serialize-javascript`:** `^7.0.5`
  matches the `first_patched_version` of the most
  stringent advisory (#6, CPU DoS). The major-version bump
  from 6 → 7 is required — there is no patched 6.x
  release — and the public API is unchanged.
- **Verification by lockfile inspection:** the acceptance
  criterion is "resolved versions in the lockfile meet
  the patched-version threshold," not "GitHub closes the
  alert." Alert auto-closure happens after merge to the
  default branch and is outside the developer's local
  control; the lockfile state is what determines whether
  the alert will close.
- **Single task:** all four alerts are remediated by one
  atomic change — two override entries plus a single
  `pnpm install`. Splitting into per-package tasks would
  produce two consecutive lockfile rewrites with no
  intermediate value.
