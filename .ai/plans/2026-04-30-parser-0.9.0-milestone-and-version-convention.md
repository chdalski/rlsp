# Bump rlsp-yaml-parser to 0.9.0 and Add Version-Edit Convention

## Goal

Establish `rlsp-yaml-parser` 0.9.0 as the next published
version on crates.io to step past the orphan 0.8.0, add a
project convention preventing agent-driven
`version = "..."` edits in any `Cargo.toml`, and yank the
orphan 0.8.0 from crates.io once 0.9.0 is live. Going to
0.9.0 (not 1.0) preserves pre-1.0 freedom while skipping
cleanly past the orphan; the convention codifies the
policy that release-plz owns version progression from
conventional commits, and milestone bumps are
user-directed; yanking 0.8.0 prevents future
`cargo add rlsp-yaml-parser` users from pinning to the
incident snapshot.

## Context

- **The 0.8.0 orphan.** On 2026-04-26, two perf commits —
  `3f15780` ("borrow constant tag URIs from resolver")
  and `d853605` ("box rare per-node metadata behind
  Option<Box<NodeMeta>>") — manually bumped
  `rlsp-yaml-parser` 0.6.0 → 0.7.0 → 0.8.0 inside
  otherwise unrelated commits. release-plz then ran on
  `main`, found 0.8.0 in `Cargo.toml`, and published
  0.8.0 to crates.io with tag `rlsp-yaml-parser-v0.8.0`.
  Commit `b96bedc` (2026-04-27) reverted `Cargo.toml`
  back to 0.6.0 with the message "release-plz manages
  versions"; release-plz then resumed normal cadence
  (0.6.0 → 0.6.1, current). 0.8.0 remains on crates.io
  as a permanent orphan version.
- **Why 0.9.0 (not 1.0).** crates.io semver progression
  needs a version greater than 0.8.0. The parser is
  spec-faithful (368/368 yaml-test-suite cases) and the
  LSP feature pull on the AST has cooled (last
  AST-shape `feat:` commit was 2026-04-23), but recent
  work has produced multiple semver-breaking shape
  changes in the last two weeks: `Span` representation
  (`716771f`), `EventMeta`/`NodeMeta` boxing (`76904a9`,
  `d853605`), `Cow<'static, str>` tag URIs (`3f15780`),
  `Schema` API removal (`073f128`), and
  `anchor_loc`/`tag_loc` field additions (`265cb5a`,
  `0ca3083`, `521e3b5`, `a7870c9`). Pre-1.0 absorbs these
  silently; 1.0 would require a 2.0 each time. The user
  has decided to defer 1.0 until a longer quiet stretch
  validates API stability.
- **Files involved.**
  - `rlsp-yaml-parser/Cargo.toml` line 3:
    `version = "0.6.1"`
  - `rlsp-yaml/Cargo.toml` line 19:
    `rlsp-yaml-parser = { version = "0.6", path = "../rlsp-yaml-parser" }`
  - `Cargo.lock` package entry for `rlsp-yaml-parser`:
    `version = "0.6.1"`
  - `CLAUDE.md` Conventions section (root) — the existing
    bullet "Automated releases via release-plz; tag
    format: `<package>-v<version>` ..." sits adjacent to
    where the new convention belongs.
- **release-plz behavior.** `release-plz.toml` is
  minimal — workspace-level changelog config, 10 min
  publish timeout, `git_tag_name = "{{ package }}-v{{ version }}"`,
  and one `[[package]]` entry for `rlsp-yaml`. release-plz
  uses the version in `Cargo.toml` as the source of
  truth: when the workspace is pushed to `main` with
  `Cargo.toml` at 0.9.0, release-plz will open a release
  PR for `rlsp-yaml-parser-v0.9.0` and (after merge)
  publish 0.9.0 to crates.io.
- **Manual milestone bumps are legitimate.** The
  convention forbids agent-driven version edits, but
  human-decided milestone bumps (this plan; future 1.0)
  are user-directed and applied as a single deliberate
  commit. The convention text must distinguish these
  cases so it reads as a process boundary, not a blanket
  prohibition.
- **Release publication is out of scope of this plan.**
  After Task 2 lands and is pushed to `main`, release-plz
  opens a release PR; the user merges it and 0.9.0
  publishes to crates.io. This plan stops at the
  push-to-main boundary — the user explicitly requested
  a pause for push and release after Task 2.
- **Yanking 0.8.0 is in scope.** The session
  environment has `CARGO_REGISTRY_TOKEN` set, so the
  lead can run `cargo yank --version 0.8.0 rlsp-yaml-parser`
  directly after the user confirms 0.9.0 is live on
  crates.io. The yank produces no git diff, so it does
  not fit the developer-reviewer pipeline; it lives in
  the "Lead Actions After Task 2" section.
- **References.**
  - [release-plz](https://release-plz.ieni.dev/) — version
    bump source-of-truth and PR workflow.
  - [Cargo manifest reference — version field](https://doc.rust-lang.org/cargo/reference/manifest.html#the-version-field).
  - [crates.io yank documentation](https://doc.rust-lang.org/cargo/commands/cargo-yank.html).

### Step A: Send push-and-release reminder

Immediately after Task 2 is committed, the lead pauses
execution and sends the user this reminder verbatim:

> Tasks 1 and 2 are committed locally. Next steps are
> yours:
>
> 1. Push `main` to the remote — `release-plz.yml` will
>    open a release PR for `rlsp-yaml-parser-v0.9.0`.
> 2. Merge that PR — release-plz publishes 0.9.0 to
>    crates.io and creates the `rlsp-yaml-parser-v0.9.0`
>    git tag.
> 3. Confirm back here when `cargo search rlsp-yaml-parser`
>    shows 0.9.0 as the latest version. I'll then yank
>    0.8.0 (the `CARGO_REGISTRY_TOKEN` is in this
>    session's environment, so I can run the yank
>    directly).

The plan stays in `InProgress` status. The lead does
not run `TeamCreate` or send any further task while
waiting for the user's confirmation.

### Step B: Verify and yank 0.8.0

Once the user confirms 0.9.0 is live, the lead executes
this sequence in order:

1. **Pre-yank check.** Run
   `cargo search rlsp-yaml-parser` and confirm the
   first listed line is `rlsp-yaml-parser = "0.9.0"`.
   If the output instead shows 0.6.1 (or any version
   other than 0.9.0) as the latest, stop and report to
   the user — do not yank, because yanking 0.8.0 while
   0.9.0 is not yet the resolved-default would leave
   0.6.1 as the highest non-yanked version.
2. **Run the yank.** Execute
   `cargo yank --version 0.8.0 rlsp-yaml-parser`. The
   command must exit 0; expected stdout is
   `Yank rlsp-yaml-parser@0.8.0`.
3. **Post-yank verification.** Query the crates.io API
   directly — `cargo search` filters yanked versions and
   cannot distinguish yanked from un-yanked, so it is
   unsuitable here. Run:

   ```
   curl -fsSL "https://crates.io/api/v1/crates/rlsp-yaml-parser/0.8.0" \
     | grep -o '"yanked":[^,]*'
   ```

   The expected output is `"yanked":true`. If the value
   is `false` or the field is missing, the yank did not
   take effect — report the failure to the user and do
   not mark the plan Completed.
4. **Mark the plan Completed.** Update the plan's
   `**Status:**` header from `InProgress` to
   `Completed (YYYY-MM-DD)`, check off all step
   checkboxes (including the two Out-of-scope steps now
   that the user has completed them and 0.8.0 is
   yanked), and commit the change with
   `docs: mark plan complete` in conventional commit
   format.

## Decisions

- **Target version is 0.9.0, not 1.0.** Going to 1.0
  would commit to semver guarantees on the parser's
  public API. Recent perf and feature work has produced
  multiple breaking shape changes in the last two weeks;
  the user has determined the API is not yet stable
  enough for a 1.0 commitment. 0.9.0 stays in 0.x,
  preserves freedom to keep optimizing, and still solves
  the immediate problem of skipping past the 0.8.0
  orphan.
- **0.7.0 / 0.8.0 are deliberately skipped.** 0.8.0 is
  the orphan on crates.io; 0.7.0 was published only as
  the intermediate state of the same incident and is also
  burned. 0.9.0 is the next clean slot above the highest
  published version.
- **Yank 0.8.0 deferred to after 0.9.0 publishes.**
  Yanking before 0.9.0 is published would leave 0.6.1 as
  the resolved-default for the gap window. Yanking after
  means 0.9.0 is already the highest non-yanked version,
  so the yank only affects users explicitly pinning to
  `"0.8"` — lower stakes.
- **Yank is a Lead Action, not Task 3.**
  `CARGO_REGISTRY_TOKEN` is set in the session
  environment, so the lead can run `cargo yank` directly.
  The yank produces no git diff and no commit, so the
  developer-reviewer pipeline (which assumes a code
  baseline and verified file list) does not apply.
  Putting the yank in the pipeline as Task 3 would
  create an empty commit or force the developer to
  invent a verification artifact. The Lead Action
  pattern is already established in this plan for the
  reminder; the yank fits the same shape — operational,
  one command, executed after a user-confirmed
  precondition.
- **No CI guard added in this plan.** A CI workflow that
  blocks PRs editing `version = "..."` lines (with a
  label or commit-marker escape hatch) was discussed and
  explicitly deferred. The CLAUDE.md convention is the
  policy layer; CI enforcement may be added later if the
  convention is violated again.
- **Convention placement.** The new bullet goes in the
  existing Conventions section adjacent to the
  release-plz bullet, not in a new section. The
  Conventions section already documents release-plz
  usage; the new bullet is the "what agents must not do"
  half of the same policy.
- **One plan covers both tasks.** The version bump and
  the convention are tightly coupled — the bump is the
  deliberate user-directed edit that the new convention
  permits as a milestone exception. Splitting into two
  plans would obscure that the bump exemplifies what the
  convention permits.
- **Push reminder is a Lead Action, not a Task.** The
  user requested an explicit reminder after Task 2.
  Reminders are lead-side communication, not developer
  implementation work, so the reminder cannot pass
  through the developer-reviewer pipeline. It lives in
  the "Lead Actions After Task 2" section as Step A,
  with verbatim text the lead must send. The plan stays
  `InProgress` until the lead has yanked 0.8.0 (Step B)
  so completion does not silently bypass either action.
